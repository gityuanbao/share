# 量化数据采集项目复盘文档

> 记录从需求分析到最终技术决策的完整思路链

---

## 一、需求起点

用户目标：**通过 AKShare 获取尽可能多的量化可用数据，存入数据库（含多模态数据如图片）**

在此之前，用户已通过横纵分析法对整个量化交易数据体系做了完整调研（见 `量化交易数据获取与存储_横纵分析报告.pdf`），本项目是该调研的落地执行层。

---

## 二、需求澄清过程（问答记录）

通过四轮问答，逐步把模糊需求收敛到可执行规格：

| 轮次 | 问题 | 用户答案 | 影响的设计决策 |
|------|------|----------|--------------|
| 第1轮 | 市场范围 | A股 + 期货 + 港股 + 宏观/另类 | 确定4大数据域 |
| 第1轮 | 时间粒度 | 混合（多种粒度都要） | 需分别处理日线/分钟线模块 |
| 第1轮 | 目标数据库 | **「不需要管数据库，保证是文本类型即可」** | 输出全部改为 CSV，不接数据库 |
| 第1轮 | 采集模式 | 一次性历史补录（Backfill） | 不需要定时调度，只需一次性脚本 |
| 第2轮 | 历史深度 | 2020年至今 | START_DATE = "20200101" |
| 第2轮 | 文本格式 | CSV，每类一文件 | 确认输出格式 |
| 第2轮 | 另类数据范围 | 宏观经济 + 财经新闻 + 公司公告 + 行业/资金流 | 确定4类另类数据 |
| 第3轮 | 成功率/耗时/体积偏好 | **「哪种成功概率最大，耗时不非常长，不超20GB」** | 触发「执行决策」环节 |

**关键转折**：第1轮用户说「不需要管数据库」，彻底简化了架构——原本准备设计 TimescaleDB + ClickHouse 分层存储，直接改为 CSV 文件输出，降低了用户的运行门槛和依赖项。

---

## 三、执行决策（三维权衡）

在用户提出「成功率最大、耗时不太长、不超20GB」之后，对每个数据模块做了三维权衡：

### 决策矩阵

| 模块 | 原始选项 | 最终决策 | 决策依据 |
|------|---------|---------|---------|
| A股日线 | 全量 5500 只 | ✅ **全量** | 接口最稳定，是核心数据，不裁剪 |
| A股分钟线 | 全量 5500 只 | ⬇️ **仅沪深300，改5分钟线** | 1分钟线全量≈80GB超限；5分钟线体积是1分钟的1/5；300只足够大多数策略 |
| 期货日线 | 全量 ~60 品种 | ✅ **全量** | 体积小（30MB），接口稳 |
| 港股日线 | 全量 ~2000 只 | ⬇️ **仅港股通约500只** | 大量港股是冷门股无流动性；港股通是主要可交易标的 |
| 宏观指标 | 全部可用 | ✅ **全取9类** | 体积极小（<5MB），不裁剪 |
| 财经新闻 | 5年 | ⬇️ **近2年（2024起）** | AKShare历史新闻接口稳定性差；2年内成功率高 |
| 公司公告 | 全量5500只 | ⬇️ **仅沪深300** | 300只可控（~2小时）；全量5500只需10小时且易报错 |
| 基本面财务 | 全量 | ⬇️ **沪深300+中证500约800只** | 平衡覆盖度和耗时 |
| 行业资金流 | 全取 | ✅ **全取** | API简单，几分钟完成 |
| 北向/融资融券 | 全取 | ✅ **全取汇总数据** | 体积极小 |

### 体积估算

```
A股日线（5500只 × 5年）     ≈ 2.2 GB
A股5分钟线（300只 × 5年）   ≈ 1.0 GB
期货主力合约日线             ≈ 0.03GB
港股日线（500只 × 5年）      ≈ 0.2 GB
宏观经济（9类）              ≈ 0.01GB
财经新闻（2年）              ≈ 0.2 GB
公司公告元信息（300只）      ≈ 0.05GB
基本面财务（800只 × 3张表） ≈ 0.2 GB
行业/板块/资金流             ≈ 0.05GB
────────────────────────────────────
合计                         ≈ 3.9 GB  ✅ 远低于20GB上限
```

## 四、技术架构设计

### 项目结构

```
akshare_collector/
├── main.py                    # 主入口，模块调度器
├── config.py                  # 全局配置（时间范围、目录、限速）
├── requirements.txt
├── utils/
│   └── helpers.py             # 日志/重试/限速/断点续传
├── modules/
│   ├── stock_info.py          # 模块1：股票基础信息
│   ├── stock_daily.py         # 模块2：A股日线
│   ├── stock_minute.py        # 模块3：A股5分钟线
│   ├── stock_fundamentals.py  # 模块4：基本面财务
│   ├── capital_flow.py        # 模块5：资金流向
│   ├── futures_daily.py       # 模块6：期货日线
│   ├── hk_daily.py            # 模块7：港股日线
│   ├── macro.py               # 模块8：宏观指标
│   ├── news.py                # 模块9：财经新闻
│   ├── announcement.py        # 模块10：公司公告
│   └── sector.py              # 模块11：行业板块行情
└── data/                      # 输出目录（自动创建）
    ├── stock/
    ├── futures/
    ├── hk_stock/
    ├── macro/
    ├── news/
    ├── announcement/
    └── sector/
```

### 四大工程原则

1. **断点续传**：`already_done()` 检查文件存在且非空，跳过已完成任务，中途重启不重跑
2. **错误隔离**：单只股票/单个模块失败 → 写 `errors.log` → 继续下一个，不中断主流程
3. **自动重试**：`@retry` 装饰器，指数退避（1s/2s/4s），最多3次
4. **请求限速**：每次API调用后 `sleep(0.4s)`，避免被AKShare源站封IP

### 数据流

```
AKShare API
    ↓ @retry（失败自动重试）
    ↓ sleep(0.4s)（限速）
pandas DataFrame
    ↓ 过滤（时间范围/空值检查）
    ↓ already_done()（断点续传）
CSV 文件（UTF-8-BOM，Excel可直接打开）
    ↓ 错误写入 errors.log
```

## 七、已知限制与后续扩展

| 限制 | 说明 |
|------|------|
| A股Tick数据 | AKShare不支持，需购买Tushare Pro或交易所数据 |
| 港股分钟线 | AKShare接口不稳定，本版本未纳入 |
| 新闻全文 | 只有标题+摘要，正文需进一步爬取 |
| 公告PDF | 只有元信息（标题/日期/类型），全文需PDF下载解析 |
| 卫星/图像数据 | AKShare不覆盖，需Planet Labs/Sentinel等专业数据源 |
| 实时数据 | 本脚本为历史补录，实时采集需另起WebSocket进程 |

**扩展路径：**
- 数据量大之后：把CSV导入 TimescaleDB（时序）或 ClickHouse（OLAP分析）
- 多模态图像：用 `mplfinance` 从CSV日线数据自动生成K线图，存为PNG
- 情绪因子：对 `news/` 目录下的文本跑 FinBERT，结果追加为新列
